# Staffbase RFP Response Engine

## 1. Trigger（起動条件）

以下の条件がそろったときにこのスキルを起動する。

- ユーザーが `.xlsx` ファイルをアップロードした
- かつ「RFP」「回答」「質問票」のいずれかのワードが言及された

---

## 2. 情報ソースの優先順位

| 優先度 | 対象 | ソース |
|--------|------|--------|
| 1（最優先） | 製品機能・仕様・技術情報 | Product Master Source（下記URL） |
| 2 | セキュリティ・ISMS・認証 | `security_kb.md` / `isms_controls.md` |
| 3 | サポート・保守・運用体制 | `ops_kb.md` |
| 4 | 法的・コンプライアンス要件 | `compliance_kb.md` |
| 常時クロスチェック | 機能制約・非対応事項 | `limitations_kb.md`（必須） |

**Product Master Source URL:**
```
https://raw.githubusercontent.com/YoheiUmezu/staffbase-product-reference/main/staffbase_urls.md
```

> ⚠️ 上記URL以外のインターネット情報・過去の学習データに基づく回答は一切禁止。
> KBファイルはすべて Project Files から参照する（references/ への複製不要）。

---

## 3. 回答生成アルゴリズム（必須手順）

各設問に対して、以下の4ステップを厳守する。

### Step 1 — 分類（Classification）

設問が以下のどちらに属するかを判定する。

- **A: 製品機能系**（機能の有無・仕様・操作性・連携など）
- **B: セキュリティ／運用／法務系**（認証・データ保護・SLA・契約など）

### Step 2 — 情報抽出（Retrieval）

**A の場合（製品機能）:**

1. Product Master Source URL にアクセスし、掲載されているURLリストを取得する
2. 設問の機能に最も合致する**個別記事URL**（`/articles/〜` を含むもの）を特定する
3. その個別記事ページの内容をリアルタイムで解析し、情報を抽出する

> 🚫 禁止：`https://support.staffbase.com/hc/ja` のようなトップページ・ディレクトリルートのみでの情報抽出。必ず末端の個別記事までドリルダウンすること。

**B の場合（セキュリティ／運用／法務）:**

- 対応するKBファイル（`security_kb.md` / `isms_controls.md` / `ops_kb.md` / `compliance_kb.md`）から情報を抽出する

### Step 3 — 制約クロスチェック（Constraint Verification）

- 「可能」と判断した場合でも、**必ず `limitations_kb.md` を確認する**
- 制約が存在する場合は「可能です」と断言せず、以下のいずれかで回答する
  - `条件付きで可能です（制約あり）`
  - `対応しておりません`

### Step 4 — フォールバック処理（Fall-back）

- 上記いずれのソースにも根拠が見つからない場合、**推測・希望的観測は一切禁止**
- 該当設問の回答欄には以下を記載する：

```
要確認（公式資料未記載）
```

- **同時に、該当設問の情報を「要確認リスト」へ追記する**（詳細は Section 5 参照）

---

## 4. 出力ルール（answered_rfp.xlsx）

### ファイル形式

- 元のExcelファイルの**構造・列・行・書式を100%維持**した `answered_rfp.xlsx` を生成する

### 回答記載ルール

各回答の末尾に、以下の形式で参照を明記する。

```
参照：https://support.staffbase.com/hc/ja/articles/〔記事ID〕
```

または

```
参照：〔KBファイル名〕（例: security_kb.md）
```

> 🚫 禁止：`参照：https://support.staffbase.com/hc/ja` のような汎用URL・ルートディレクトリのみの記載。**必ず具体的な記事IDを含むフルURLを記載すること。**

### 表現基準

| 条件 | 表現 |
|------|------|
| 公式ソースに根拠あり・制約なし | `可能です` |
| 公式ソースに根拠あり・制約あり | `条件付きで可能です（〔制約の概要〕）` |
| 標準機能か追加オプションかが明確 | その旨を明記する |
| 根拠なし | `要確認（公式資料未記載）` |

- **文体：** 丁寧なビジネス日本語（です・ます調）
- **禁止表現：** 「できると思われる」「対応予定」「〜と考えられます」など根拠のない表現

---

## 5. フォールバック後の処理（要確認リスト出力）

### 出力ファイル

`answered_rfp.xlsx` の生成と**同時に**、以下のテキストファイルを生成する。

**ファイル名：** `要確認リスト.txt`

### フォーマット

```
====================================================
Staffbase RFP 要確認リスト
生成日時：〔YYYY-MM-DD〕
====================================================

以下の設問は、公式資料および提供KBファイルに根拠が見つからなかったため、
「要確認（公式資料未記載）」と回答しています。
社内確認またはStaffbase担当者への問い合わせをお願いします。

----------------------------------------------------
No.〔連番〕
設問番号：〔ExcelシートのRow番号またはID〕
設問内容：〔設問のテキストをそのまま転記〕
理由　　：根拠となる公式記事またはKBファイルが見つからなかった
----------------------------------------------------
（該当設問ごとに繰り返す）

====================================================
合計：〔件数〕件
====================================================
```

### 出力条件

- 「要確認」が**0件の場合**：`要確認リスト.txt` には「要確認項目はありませんでした。」とのみ記載して出力する
- 「要確認」が**1件以上の場合**：全件を上記フォーマットで記載して出力する

---

## 6. 禁止事項（厳守）

| 禁止事項 | 詳細 |
|----------|------|
| 一般論の流用 | SlackやMicrosoft Viva等、他SaaS製品の知識を流用しない |
| 曖昧な表現 | 「できると思われる」「対応予定」等、根拠のない記述を禁止 |
| 汎用URLの参照記載 | `/articles/〜` を含まないURLのみの参照記載を禁止 |
| 外部検索の利用 | 指定URL・KBファイル以外のインターネット情報を回答に含めない |
| 推測による補完 | 根拠不明の場合は必ず「要確認」とし、推測で埋めない |
